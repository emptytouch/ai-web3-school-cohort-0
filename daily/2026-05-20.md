## 学习笔记

### 核心概念

**1. Prompt 是软约束，不是安全边界**
Prompt 能提高模型遵循任务的概率，但不能保证模型永远按规则行动。真正的边界必须由代码、权限、校验和审计来承担。指令分层要清楚：系统规则、开发者规则、用户目标、检索内容不能混在一起。

**2. Context 是信息治理问题**
模型只能基于它看见的上下文行动。Context 不是简单的长文本拼接，而是为每类信息标注来源、时效、权限和可信度。可信来源要显式标注，上下文要可刷新，记忆要可撤销。

**3. 结构化输出是连接模型和系统的桥梁**
关键结果尽量用 JSON schema、函数参数或明确字段承载。结构化输出不是为了好看，而是为了让后续代码能检查、拒绝、记录和回归测试。

### 关键术语

| 术语 | 一句话理解 |
|------|-----------|
| Instruction | 给模型的任务规则：角色、目标、禁止行为、输出格式 |
| Few-shot | 在 prompt 里放少量示例，让模型模仿判断方式和输出格式 |
| Structured Output | 让模型按固定结构返回结果（JSON/schema），便于代码处理 |
| Prompt Injection | 攻击者通过外部内容让模型忽略原始规则或执行危险操作 |
| Context Window | 模型一次请求能处理的最大上下文范围 |
| Context Engineering | 设计上下文进入模型的方式：选择、排序、裁剪、标注来源 |
| Memory | 跨请求保留的信息（用户偏好、历史任务），但不能替代实时授权 |
| Knowledge Base | 系统可检索的外部知识库，需要维护来源和时效 |

### Prompt 在 AI × Web3 系统中的位置

Prompt 处在用户目标和模型行为之间。但 prompt 不应该独自承担安全。更稳的链路是：
1. Prompt 定义任务和输出格式
2. Context 提供可信数据和来源边界
3. Model 生成解释或候选动作
4. Code 校验 schema 和业务规则
5. Guard / simulation 检查链上影响
6. Human check 确认高风险动作

### Context 分层设计

一个靠谱的 Agent 上下文通常分五层：
- **指令层**：系统规则、工具使用规则、禁止事项
- **任务层**：用户目标、本次会话参数
- **事实层**：链上状态、工具结果、simulation
- **知识层**：文档、标准、论坛、历史案例
- **记忆层**：用户偏好和项目配置

层级越清楚，越容易做权限控制、prompt injection 防护和审计。

## Agent 章节笔记

### 核心概念

**1. Agent 不是自主权本身，而是被约束的执行循环**
Agent 的关键不在"像人"，而在于能不能在明确权限和可审计流程里把建议推进到行动。目标、工具、状态、权限和停止条件缺一不可。一个只有模型和工具的 Agent 只是"会说话的脚本执行器"。

**2. 工具比回答更危险——按风险等级分层**
读数据、写数据库、发送请求、改配置、触发支付，每个操作的风险等级完全不同。工具设计必须明确：输入 schema、权限范围、是否只读、是否产生外部副作用、调用前后如何记录、哪些需要人工确认。

**3. State 必须外置和可查，不能只藏在模型上下文里**
任务进度、工具结果、失败原因、用户确认都要记录到可查询、可恢复、可审计的存储中。否则无法回答：Agent 为什么调用了这个工具？是否已经拿到用户确认？哪一步开始偏离目标？

**4. 停止条件要明确**
达到目标、超出预算、信息不足、风险越界、用户拒绝——都应该让 Agent 停下来。不能只有"继续"的逻辑，没有"停止"的逻辑。

**5. Reflection 不能替代外部验证**
Agent 自我反思仍然由模型完成，模型可能给自己的错误找理由。写入、授权、支付这类动作，reflection 只能作为辅助诊断，不能作为最终安全判断。自我检查可以提高质量，确定性检查才能承载风险。

### AI × Web3 Agent 的安全架构

一个相对稳的 AI × Web3 Agent 架构：
1. 用户提出目标和约束
2. Agent 读取上下文并生成计划
3. 系统把计划拆成只读步骤和候选写入步骤
4. 只读工具自动执行，写入工具进入 policy 检查
5. Simulation 展示链上影响
6. 用户确认高风险动作
7. Wallet / Smart Account 执行
8. 日志记录每一步和最终状态

**最危险的设计**：让 Agent 同时拥有模糊目标、广泛工具、长期记忆和大额资产权限。

### 关键术语

| 术语 | 一句话理解 |
|------|-----------|
| Tool Use | Agent 调用外部能力（API、数据库、支付等），从"会回答"变成"能做事" |
| Planning | 把目标拆成步骤，模型生成的计划只是候选路线，不是授权 |
| State | Agent 当前任务状态（目标、步骤、工具返回、错误、确认记录），必须外置可查 |
| Reflection | Agent 检查自己的中间结果并修正，但不能替代外部验证 |
| Multi-Agent | 多个 Agent 分工协作，适合复杂工作流但会放大协调问题 |
| ReAct | Reasoning + Acting 的交替执行模式：思考→行动→观察→思考 |
| Guardrails | 对 Agent 行为的硬性限制和检查规则，防止越权执行 |
| Simulation | 在执行前模拟操作结果（如链上交易模拟），用于风险评估 |

### 最小实践：DAO 提案研究 Agent

**只读版本**（当前阶段）：
- 读取提案正文
- 检索论坛讨论
- 总结支持和反对理由
- 标记缺失信息
- 输出投票前检查清单

**关键原则**：不要让它直接投票。要求输出里明确：
- 用了哪些来源
- 哪些结论证据不足
- 是否发现治理或资金风险
- 如果用户要投票，还需要人工检查什么

**权限升级版本**（后续）：只有当用户明确授权，并经过投票交易 simulation 后，系统才允许生成投票交易草稿。

### 思考

- 当前用 Claude/Cursor/Qoder 时，本质上是在用"单次回答"模式。如果要升级到 Agent 模式，需要给模型配备工具（查链上数据、解码交易、模拟执行）和状态管理。
- 在 Web3 场景里，Agent 的"停止条件"尤其重要——一笔交易上链就是不可逆的，必须在执行前设置足够的检查点。
- Multi-Agent 在 Web3 场景的风险：一个 Agent 的错误可能被另一个 Agent 当成事实传递，导致错误扩散。先从单 Agent + 多步骤开始，比直接上 Multi-Agent 更稳。
