## 学习笔记

### 一、检索增强生成（RAG）

#### 核心概念

**RAG 的本质是证据链，不是向量库。**
模型训练知识会过期，context window 也装不下所有文档。RAG 的流程是：用户提问 → 从知识库检索相关材料 → 让模型基于材料回答。关键是"答案有来源、有版本、有边界"。

**第一次判断比模型生成更重要：**
- Chunking：文档怎么切？按结构切（标题/API/函数说明）比按字数切更稳
- Retriever：检索时不只看语义相似度，还要过滤来源、版本、时间、可信等级
- Rerank：候选多时重排序，把更相关更完整的排前面
- Citation：关键结论必须能追溯到具体文档段落，没有 citation 的 RAG 只是把幻觉搬到了检索系统里

#### 关键术语

| 术语 | 一句话理解 |
|------|-----------|
| Chunking | 把长文档切成可检索片段，按结构切优于按字数切 |
| Vector DB | 存储 embedding + metadata（来源/版本/可信等级），检索时先过滤再排序 |
| Retriever | 根据问题取回候选材料，可以是向量/关键词/混合检索 |
| Rerank | 对初步检索结果重新排序，提升精度，代价是延迟和成本 |
| Citation | 答案中关键结论能追溯回具体来源文档或链上记录 |

#### 在 AI × Web3 中的位置

RAG 位于 Knowledge Base 和模型之间，帮 Agent 查资料、补上下文、引用证据。包括：协议文档问答、合约接口解释、治理提案摘要、审计报告检索、交易解释时补充项目背景。

**注意**：当 RAG 结果要影响链上动作时，还需要接 simulation、policy 和 human check。文档说"可以调用"不等于用户"应该签名"。

### 二、智能体（Agent）

#### 核心概念

**Agent 不是自主权本身，而是被约束的执行循环。**
目标、工具、状态、权限和停止条件缺一不可。只有模型和工具的 Agent 只是"会说话的脚本执行器"。

**工具按风险等级分层：**
- 读数据（查余额/交易）→ 可自动执行
- 写操作（发交易/授权）→ policy 检查 + 人工确认
- State 必须外置可查，不能只藏在模型上下文里
- 停止条件要明确：目标达成/预算超限/信息不足/风险越界/用户拒绝 → 都应该停下来

**Reflection 不能替代外部验证：**
Agent 自我反思仍然由模型完成，模型可能给自己的错误找理由。写入、授权、支付这类动作，reflection 只能辅助诊断，不能作为最终安全判断。

#### AI × Web3 Agent 安全架构

1. 用户提出目标和约束
2. Agent 读取上下文并生成计划
3. 系统把计划拆成只读步骤和候选写入步骤
4. 只读工具自动执行，写入工具进入 policy 检查
5. Simulation 展示链上影响
6. 用户确认高风险动作
7. Wallet / Smart Account 执行
8. 日志记录每一步和最终状态

**最危险的设计**：模糊目标 + 广泛工具 + 长期记忆 + 大额资产权限。

#### 关键术语

| 术语 | 一句话理解 |
|------|-----------|
| Tool Use | Agent 调用外部能力，从"会回答"变成"能做事" |
| Planning | 模型生成的计划只是候选路线，不是授权 |
| State | 任务状态外置可查（进度/工具返回/错误/确认记录） |
| Reflection | Agent 自我检查中间结果，但不能替代外部验证 |
| Guardrails | 对 Agent 行为的硬性限制和检查规则 |

### 三、框架（Frameworks）

#### 核心概念

**框架是系统边界的表达，不是智能本身。**
先理解工作流，再决定用不用框架。框架选错，问题通常不是"跑不起来"，而是后面调不动、测不了、换不掉。

**判断框架的三个原则：**
- 简单链路保持简单：一次模型调用 + 一次检索 + 一次格式化，不一定需要复杂 Agent 框架
- 长流程需要显式 state：多步任务/工具调用/人工确认/失败恢复，应该有可查询的 state
- 框架要能退出：如果很难换模型/换向量库/换部署方式，长期成本会很高

#### 主要框架对比

| 框架/工具 | 适合场景 | 注意事项 |
|----------|---------|---------|
| LangChain | 快速把模型能力接到外部系统，学习 AI 应用常见组件 | 小心"抽象太早"，没搞清楚工作流就套 chain/agent 后面排查难 |
| LangGraph | 多步工具调用、重试、人工确认、分支、恢复、有状态 Agent | 单次问答场景太重，任务需要追踪"走到哪一步"时再考虑 |
| OpenAI Agents SDK | 带 tool/handoff/guardrails/tracing 的 Agent 应用 | SDK 帮你执行流程，但要你定义哪些工具可用、哪些动作需要确认 |
| DSPy | 有明确数据集 + 评估指标的可重复任务（分类/抽取/问答） | 不要只靠感觉调 prompt，让任务/数据/指标进入系统 |

#### 在 AI × Web3 中的分工

- **AI Framework** 管理 prompt、tools、state、eval 和 trace
- **Web3 基础设施** 管理账户、签名、合约、交易和链上状态
- **产品层** 定义用户目标、权限边界、确认流程和失败处理

框架可以组织 Agent，不能替用户承担资产风险。

### 四、氛围编程（Vibe Coding）

#### 核心概念

**Vibe Coding 不是"把需求丢给 AI 然后等代码出现"。**
它是一种人和 AI Coding Agent 共同迭代软件的工作方式：人负责方向、约束和验收，Agent 负责生成、修改、搜索和执行工程动作。

**AI Coding 的核心是把工程反馈循环压短：**
- 任务要小：越具体越有边界，Agent 越容易产出可审查结果
- 上下文要准：让 Agent 看对文件、设计约束和错误输出，比写长篇需求更重要
- 验证要硬：运行测试、类型检查、构建、截图或日志，比"看起来对"可靠

#### 适用 vs 不适用

✅ 适合：搭原型、修小 bug、补测试、写脚本、重构局部模块、解释陌生代码
❌ 不适合：无边界地"重写整个项目"，任务范围太大时 Agent 容易改到不该改的地方

#### Repo Workflow 中的 AI Coding

不要让 AI 绕过 repo 流程，更好的方式是让它**参与**流程：
- 从 issue 提炼任务边界
- 搜索相关文件
- 生成最小 patch
- 跑测试并解释失败
- 补充 changelog 或文档
- 写 PR summary 和验证说明

#### 在 AI × Web3 中的注意

AI × Web3 项目通常有前端、合约、脚本、索引器、Agent 后端和文档。Vibe Coding 可以显著加快探索速度，**尤其适合 hackathon 和早期原型**。

但合约、签名、权限、支付、迁移脚本不能只靠 Agent 生成后直接上线。高风险动作必须经过审查、模拟和多方确认。

#### 关键术语

| 术语 | 一句话理解 |
|------|-----------|
| Vibe Coding | 人和 AI Coding Agent 共同迭代软件，人管方向验收，Agent 管生成执行 |
| Claude Code / Codex CLI | 把 AI Agent 放进本地开发环境，能读文件/改文件/跑命令 |
| OpenCLI | 把网站/浏览器/本地工具统一成 Agent 可调用的 CLI 接口 |
| Repo Workflow | AI 参与 issue→branch→commit→test→review→merge 流程 |

1. **RAG 的 Chunking 策略在 Web3 场景怎么定？** 🤔
   - Solidity 文档/审计报告/EIP 标准各不相同
   - 函数说明和参数表需要保持完整，不能切碎
   - 链上数据（交易日志）怎么切 chunk？

3. **Vibe Coding 在合约开发中的安全边界？** 🤔
   - Agent 可以写测试、解释 ABI、生成脚本
   - 但合约主逻辑能不能让 Agent 生成？
