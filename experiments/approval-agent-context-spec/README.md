# 钱包授权检查 Agent — Context Spec 实验

## 场景
用户问："这个 dApp 要我 approve，可以签吗？"

## 实验目标
设计一份完整的上下文规范，明确模型在回答前必须拿到哪些数据，以及哪些数据可信、哪些不可信。

## 文件说明

| 文件 | 内容 |
|------|------|
| `context-spec.md` | 完整的上下文规范（字段清单、分层、风险规则、输出格式） |
| `README.md` | 本文件，实验说明 |

## 核心设计

### 四层上下文

1. **实时查询**（8 个字段）— 链上直接查，任何失败 → 停止
2. **可缓存**（4 个字段）— TTL 5-15 分钟，未命中 → 标记未知
3. **用户输入**（3 个字段）— 标记为不可信，仅交叉验证
4. **禁止当成事实**（4 类）— dApp 声明、模型印象、社交推荐、用户评价

### 7 条确定性风险规则

| 条件 | 等级 |
|------|------|
| 无限授权 | high |
| spender 未验证 | high |
| spender 黑名单 | critical |
| URL 不匹配 | critical |
| 授权超余额 | medium |
| 授权叠加 | medium |
| 新合约 < 7 天 | medium |

## 实验结论

1. 链上实时数据是唯一可信来源
2. 失败即停止（无法获取 = 不建议签名）
3. 确定性规则兜底，不依赖模型推断
4. 不可信标记防止 prompt injection
5. context_used 字段保证可审计
