## 学习笔记

### 核心概念

**1. LLM 是概率模型，不是数据库**
LLM 生成的是"概率上合理的输出"，而不是天然可信的事实。它擅长理解、归纳、生成和规划，但不能单独承担事实来源、权限判断和最终执行。关键事实必须来自数据库、API、日志等可信系统。

**2. Token 是模型的基本处理单位**
Token 不等于一个汉字或一个英文单词，而是 tokenizer 切分后的片段。它直接影响：上下文能放多少、调用成本是多少、模型能不能完整看见关键信息。代码、JSON、长标识符比普通段落更吃 token。

**3. Hallucination（幻觉）是系统性风险**
模型会生成看起来合理但不真实的内容。在普通问答里只是质量问题，但在带执行能力的系统里会变成流程风险——错误参数、错误权限解释、错误操作建议可能进入自动化链路。处理幻觉不能只靠更好的 prompt，需要外部校验：来源引用、schema 校验、规则检查、人工确认。

### 关键术语

| 术语 | 一句话理解 |
|------|-----------|
| Token | 模型切分文本的最小片段，影响上下文长度和成本 |
| Embedding | 把文本映射成向量，用于语义搜索和 RAG |
| Transformer | 现代 LLM 的核心架构，通过 attention 机制学习上下文关系 |
| Hallucination | 模型生成了看起来合理但不真实的内容 |
| Multimodal | 能处理文本、图片、音频等多种输入的模型 |

### LLM 在 AI × Web3 系统中的位置

LLM 位于**理解和生成层**，负责：
- 把用户目标转成可讨论的计划
- 把复杂链上数据解释成人能读的语言
- 把文档和代码串成可执行思路

但它需要配合其他层：
- **数据层**：RPC、索引器、预言机、向量库
- **编排层**：Prompt、Context、RAG、Agent workflow
- **执行层**：工具调用、钱包、合约交互
- **安全层**：Guard、simulation、权限策略、日志

> LLM 越靠近执行层，越需要把自然语言输出变成可验证对象。

### 疑问与思考

1. **Token 限制的实际影响** ✅ 已讨论
   - 核心思路：不要塞全部数据，让模型按需取数
   - 最佳方案：工具调用（getTransaction/getABI/decodeLog）+ 按需返回
   - 备选：预处理压缩、RAG 检索增强
   - 实践建议：给 Agent 配工具函数，比贴数据高效得多

2. **幻觉检测的自动化** ✅ 已讨论
   - 可以部分自动化，无法 100%
   - 方案：交叉验证（Dual Model）、来源引用校验、Schema 校验、确定性规则兜底
   - 实践建议：合约参数用 schema 校验 + 大额交易强制人工确认

3. **Embedding 在 Web3 场景的应用** ✅ 已讨论
   - 完全可以，且很有价值
   - 场景：相似合约搜索、交易模式检索、文档知识库、安全审计辅助
   - 实践建议：先把 Solidity 文档/EIP 标准做 embedding，开发时自然语言搜索
