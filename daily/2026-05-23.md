## 今日完成：遗留实验逐一完成

### 1. MCP Server 只读版（experiments/mcp-server/）

实现了零依赖的 MCP stdio server，暴露两个工具：
- `search_docs(query)` — 在白名单目录中搜索文档
- `get_file(path)` — 读取白名单目录中的文件

安全机制：路径白名单校验、文件大小限制（1MB）、跳过 node_modules、调用日志（JSON Lines 格式）。

测试通过：initialize / tools/list / search_docs / get_file 均正常。已注册到 ~/.hermes/config.yaml 的 mcp_servers。

### 2. 交易解释器 Eval 30 条样本（experiments/tx-risk-prompt/eval-samples.md）

30 条测试样本，4 组：
- A 组（10 条）：普通转账、swap、NFT 购买、质押、精确授权、取消授权、多签、桥接、DAO 投票
- B 组（10 条）：意图不匹配、授权超余额、授权叠加、新合约、simulation 失败、gas 异常、转合约地址、multicall、跨链、合约升级
- C 组（5 条）：历史攻击模式 — 无限授权钓鱼、闪电贷、假代币、重入攻击特征、地址欺骗
- D 组（5 条）：Prompt Injection — dApp 声明注入、user_intent 覆盖、伪造 simulation、双关语义、超长输入

每条定义：输入、期望行为、必须包含、必须拒绝/提醒、引用来源。

### 3. Chain-aware Context 数据结构（experiments/agent-experiments/chain-aware-context-schema.md）

7 层上下文结构：
1. chain — 链状态（block_number, gas_price, sync_status）
2. account — 账户状态（balance, nonce, token_balances, allowances）
3. transaction — 交易解析（function decode, params, batch 拆解）
4. contract_context — 合约信息（验证状态、年龄、审计、黑名单）
5. simulation — 模拟结果（asset_changes, events, warnings）
6. external_signals — 辅助信号（价格、文档、历史交互）
7. untrusted_inputs — 不可信输入隔离层

每层有 source / query_status / failed_fields，支持 degraded 模式。

### 4. Web3 Tool Use Schema（experiments/agent-experiments/web3-tool-use-schema.md）

9 个工具：
- T1-T8 只读：eth_getBalance / eth_call / eth_getTransactionByHash / eth_getTransactionReceipt / eth_estimateGas / eth_getCode / eth_blockNumber / alchemy_simulate
- T9 写入：eth_sendRawTransaction（带安全门控：simulation_pass + user_confirmation）

统一错误码、raw+human 双格式返回、3 种调用模式（分析已有交易/分析待签名/执行交易）。

### 5. Agent Workflow 状态机（experiments/agent-experiments/agent-workflow-statemachine.md）

9 个状态：S0_ABORT → S1_INPUT → S2_FETCH → S3_PARSE → S4_RETRIEVE → S5_SIM → S6_CHECK → S7_OUTPUT → S8_EXEC → S9_CONFIRM

关键设计：
- S6_CHECK 分 Phase A（确定性规则）+ Phase B（LLM）+ Phase C（冲突解决取高）
- S6_HUMAN 和 S8_EXEC 双重人工检查点
- 每个状态有降级路径（S3_DEGRADE / S5_DEGRADE）
- workflow_trace 全程可追溯

## 关键收获

1. **MCP server 本质是 stdio 上的 JSON-RPC** — 读 stdin 行，解析请求，写 stdout 响应。不需要 SDK 也能手写。
2. **Chain-aware Context 的核心是分层 + 降级** — 实时数据必须，缓存数据可选，不可信数据隔离。每层独立失败不影响整体。
3. **确定性规则 > LLM 推断** — 安全关键判断（无限授权、黑名单、地址不匹配）用规则兜底，LLM 只做补充分析。
4. **状态机的价值在于可追溯** — workflow_trace 让 Agent 的每一步决策都有据可查。

## 疑问与思考

1. **MCP server 的 search_docs 目前是线性搜索，文件多了怎么办？**
   → 后续可加倒排索引或向量索引。当前阶段够用。

2. **Chain-aware Context 的 TTL 怎么定？**
   → 链状态 30s，合约信息 15min，价格 5min。不同数据有不同的时效性要求。

3. **Agent Workflow 的 S6_HUMAN 怎么实现异步等待？**
   → CLI 模式：阻塞等待用户输入。Web 模式：返回 pending 状态 + webhook/polling。
