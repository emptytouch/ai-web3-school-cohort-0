## 学习笔记

### 一、模型上下文协议（MCP）

#### 核心概念

**MCP 是 AI 应用的"工具接口标准"。** 它解决的不是模型聪不聪明，而是模型如何以可描述、可复用、可管理的方式使用外部能力。

**关键不是"能接更多工具"，而是让工具接入变得可描述、可发现、可限制。** Server 暴露了什么能力、输入 schema 是什么、返回什么结果、哪些动作有副作用，都应该被清楚表达。

#### 核心组件

| 组件 | 职责 | 关键设计点 |
|------|------|-----------|
| Server | 暴露 resources/tools/prompts | 边界控制：哪些只读、哪些有副作用、参数 schema、错误返回、授权、审计 |
| Client | 连接模型和 server，转发调用 | 让用户知道连了哪些 server、能调哪些工具、调用前是否需确认 |
| Tool Schema | 描述工具名字/用途/参数/返回值/约束 | 模糊的 schema 导致模型用错误参数填空 |
| Permission | 区分只读/写入、会话/长期授权、是否需要确认、是否可撤销审计 | MCP 方便≠安全，权限模型要独立设计 |

#### 在 AI × Web3 中的位置

MCP 可作为 Agent 连接链上工具和开发工具的接口层（读取区块浏览器、查询合约文档、调用 RPC、生成交易草稿）。但 **MCP 本身不是钱包安全方案**——它标准化工具入口，不能替代账户权限、交易模拟、签名确认和审计日志。

#### 最小实践

做一个只读 MCP server，暴露两个工具：
- `search_docs(query)`：搜索本地项目文档
- `get_file(path)`：读取白名单目录里的文件

要求：不能读白名单外路径、返回带来源路径、调用写日志、错误明确返回。

### 二、评估（Evaluation）

#### 核心概念

**Evaluation 是把"感觉效果不错"变成"系统可持续改进"的方法。** 没有 eval，prompt/RAG/Agent 的变化只能靠主观试用判断，迟早被回归问题拖住。

**不能被重复测量的 AI 行为，就不能被稳定改进。**

#### 评估体系

| 概念 | 一句话理解 | 关键点 |
|------|-----------|--------|
| Harness | 运行 eval 的框架：喂样本→调用系统→收集输出→运行 grader→记录结果 | 可重复是核心价值 |
| Golden Set | 精心挑选和标注的测试样本集 | 30-100 条高质量样本比一堆随便收集的问题更有用；每修一个 bug 应加入 regression set |
| LLM-as-Judge | 用模型给模型输出评分 | 适合开放式答案，但也会偏、会漏；可自动判断的字段用规则，开放式用 LLM judge，高风险保留人工抽检 |
| Regression | 防止旧问题复发 | "修 A 坏 B" 是 AI 应用常态，每次改动都重跑 regression set |
| Observability | 线上观察系统行为 | eval 在发布前，observability 在真实使用中；记录输入/检索/工具调用/输出/错误/反馈/成本/延迟 |

#### 在 AI × Web3 中的位置

AI × Web3 系统中 eval 更重要，因为错误可能影响资产、权限、治理判断和链上执行。需要特别评估：交易解释准确性、风险提示漏报、工具调用参数越界、是否能拒绝不确定请求、是否能识别 Prompt Injection、引用是否可追溯、高风险动作是否要求 human check。

#### 最小实践

准备 30 条样本（10 正常 + 10 边界 + 5 历史 bug + 5 恶意注入），每条定义：输入、期望行为、必须包含的信息、必须拒绝/提醒的情况、是否需要引用来源。每次改 prompt/模型/检索策略前后跑一遍。

### 三、微调（Fine-tuning）

#### 核心概念

**Fine-tuning 不是"模型效果不好就训练一下"。** 它适合让模型更稳定地学习某种格式、风格、领域任务或行为模式，但不适合用来补实时知识、修权限边界或替代评估。

**微调可以让模型更像你的数据，但它不会自动带来事实正确、权限安全、引用可靠或工具调用安全。** 训练进去的偏差，也会更稳定地被模型复现。

#### 微调前先问

1. 是 prompt 不清楚吗？
2. 是上下文缺了吗？
3. 是检索没拿到正确资料吗？
4. 是输出格式没有 schema 吗？
5. 是模型本身能力不够吗？
6. 是否已经有 eval 证明问题稳定存在？

#### 核心方法

| 方法 | 一句话理解 | 适用场景 | 注意 |
|------|-----------|---------|------|
| SFT（监督微调） | 用输入+期望输出样本让模型学习某类任务的回答方式 | 固定格式输出、特定语气风格、领域术语、工具调用样式 | 对数据质量非常敏感，坏数据会把坏习惯训练得更稳定 |
| LoRA（低秩适配） | 只训练较小的适配参数，降低训练成本和显存 | 资源有限、想快速试验特定任务 | 降低实验成本，但不是魔法，任务定义/数据质量/评估方法仍决定效果 |
| PEFT（参数高效微调） | 不重新训练整个模型，通过较小参数调整适配任务 | 模型较大、完全微调成本太高 | LoRA 是 PEFT 的一种 |

#### 关键原则

- **先有 eval，再微调**：否则不知道微调后到底变好了还是只是在少数样本上变顺了
- **数据质量 > 数据数量**：坏数据会把坏习惯训练得更稳定
- **实时状态/价格/文档更新/用户数据更适合放在检索或工具里**，而不是训练进模型

### 四、推理服务（Inference）

#### 核心概念

**Inference 层决定了模型能力如何被产品消费。** 它连接上游的模型/prompt/RAG/Agent，也连接下游的接口/队列/缓存/监控和用户体验。

**同一个模型，在 API、GPU 服务、本地量化模型和边缘设备上的表现会完全不同。** 你选择的不是单一模型，而是一套延迟、成本、质量、隐私和运维复杂度之间的平衡。

#### 部署方式对比

| 方式 | 优势 | 劣势 | 适用场景 |
|------|------|------|---------|
| API Model（云端托管） | 上手快、模型更新快、基础设施负担低 | 速率限制、超时、隐私数据外送、账单控制、版本变更 | 快速原型、非隐私敏感、流量波动大 |
| Local Model（本地运行） | 隐私可控、可离线、深度定制 | 吃硬件、部署复杂、上下文窗口/并发/量化方式受限 | 隐私要求高、成本敏感、特定任务（分类/抽取/代码补全/轻量 Agent） |

#### 关键概念

- **Quantization（量化）**：把模型权重精度降低（FP16→INT8→INT4），用更少显存运行大模型。代价是可能降低输出质量，尤其是长推理/代码生成/多语言/数学/工具调用场景。**判断量化模型是否可用，要用你自己的任务样本测试，不能只靠主观聊天体验。**
- **Serving**：把模型变成稳定服务——并发、队列、流式输出、缓存、监控、失败处理、版本管理。Agent 场景一次用户请求可能触发多轮模型调用，成本和延迟会被放大。

#### 在 AI × Web3 中的位置

Agent 场景下，一次用户请求可能触发多轮模型调用（读链上数据→检索文档→生成解释→模拟交易），推理层的延迟和成本设计直接影响用户体验。隐私敏感场景（如本地密钥管理）可能需要本地模型承担特定子任务。

### 五、Web3 基础概览

> 今日快速浏览 Web3 基础全貌，后续逐章深入。

#### 密码学（Cryptography）

**Web3 的账户系统把"谁有权限"从中心化数据库转成了可验证的数学关系。** 好处是开放、可组合、无需平台许可；代价是用户和应用都要承担更明确的密钥与签名风险。

| 概念 | 一句话理解 | 关键风险 |
|------|-----------|---------|
| Hash | 把任意数据变成固定长度指纹，验证数据一致性 | 哈希不是加密，不能还原原文 |
| Public Key | 可公开的密钥，用于验证签名和生成地址 | — |
| Private Key | 保密的密钥，用于签名和控制资产 | **丢失=失去控制权，泄漏=别人获得控制权** |
| Signature | 用私钥对数据签名，证明授权 | 签名内容必须能被人读懂，不能只展示难懂数据 |
| Merle Tree | 让大量数据可以被高效验证的数据结构 | — |

**理解密码学基础，是为了知道什么时候不能"点确认"。**

#### 钱包（Wallet）

钱包不是账号系统，是密钥管理工具。核心概念：助记词→私钥→公钥→地址的派生关系；签名=授权；不同链的地址格式；硬件钱包 vs 软件钱包 vs 智能合约钱包。

#### 智能合约（Smart Contract）

部署在链上的程序，定义资产规则和交互逻辑。核心概念：Solidity、合约部署不可变、upgradeable proxy pattern、gas、事件日志、ABI。

#### 开发栈（Dev Stack）

| 层级 | 工具/框架 |
|------|----------|
| 合约开发 | Solidity, Foundry, Hardhat |
| 前端 | ethers.js, wagmi, viem |
| 索引 | The Graph, ponder |
| 测试网 | Sepolia, Goerli |
| 部署 | Alchemy, Infura |

#### 网络（Network）

Ethereum L1、L2（Optimistic Rollup vs ZK Rollup）、侧链、跨链桥。核心概念：区块/ gas/ 确认数/ 最终性/ 重组风险。

#### 账户抽象（Account Abstraction）

用智能合约钱包替代 EOA，实现社交恢复、批量交易、session key、gas 代付等。核心概念：ERC-4337、UserOperation、Bundler、Paymaster。

#### 去中心化金融（DeFi）

AMM、借贷、稳定币、衍生品、闪电贷。核心概念：TVL、滑点、无常损失、清算。

#### 预言机（Oracle）

把链下数据送上链。核心概念：Chainlink、价格喂价、VRF、CCIP。

#### 索引（Indexing）

从链上事件提取数据供应用查询。核心概念：The Graph subgraph、事件监听、实时 vs 历史数据。

#### 安全（Security）

重入攻击、闪电贷攻击、权限管理、审计、bug bounty。核心概念：常见攻击向量、安全开发实践。

## 疑问与思考

1. **MCP 和 Function Calling 的边界在哪？** 🤔
   - Function Calling 是模型内置的工具调用能力
   - MCP 是外部工具接入的标准化协议
   - 两者可以互补：MCP 负责工具发现和 schema 标准化，Function Calling 负责模型实际调用

2. **Evaluation 在 Web3 场景的特殊挑战？** 🤔
   - 链上数据实时变化，golden set 可能很快过期
   - 交易解释的"正确答案"可能不唯一
   - 如何评估"不确定时拒绝回答"的能力？

3. **Fine-tuning 对 Web3 场景的价值？** 🤔
   - 可能适合：固定格式的交易解释输出、特定领域的术语规范化
   - 不适合：实时价格/余额、合约 ABI（变化太快）
   - 结论：RAG + 工具调用 > fine-tuning 用于 Web3 场景

4. **Web3 基础的学习策略？** 🤔
   - 已有 DApp demo 开发经验，很多概念不陌生
   - 重点补：账户抽象、安全攻击向量、索引
   - 可以结合已有项目边学边练
